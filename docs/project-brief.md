# Project Brief

## 1. Project Type

- Type: website
- Why this project exists: 운영 중인 마켓플레이스에 등록할 AI 웹 제작 플러그인의 소개 사이트와 도입 설명 흐름을 만든다.

## 2. Business Goal

- 마켓플레이스 방문자가 플러그인의 용도와 워크플로우를 빠르게 이해하게 만든다.
- 설치 요청, 도입 문의, 파일럿 논의 시작을 유도한다.
- KPI는 소개 페이지 방문 대비 설치 요청 전환율, 데모 요청 수, 첫 이해까지 걸리는 시간으로 둔다.

## 3. Audience

- Primary audience: AI 워크플로우 도입을 검토하는 팀 리드, 제품 담당자, 그로스 실무자
- Secondary audience: 디자이너, 프론트엔드 개발자, 내부 운영팀
- Current pain points: 기획, 디자인, 개발이 분절되어 있고 AI 도구를 써도 전체 흐름은 느리다.

## 4. Offer or Core Experience

- Product or service: Claude Code 기반 AI 웹 제작 플러그인
- Core value proposition: 브리프 입력만으로 구조 설계, Figma handoff, 구현 계획까지 이어지는 재사용 가능한 워크플로우를 제공한다.
- Main proof points or differentiators: webpage, website, webservice 전반 지원, Figma MCP handoff 포함, 기존 코드베이스 유무에 따라 동작 분리

## 5. Scope

- Required pages or screens: 홈, 기능 설명, 워크플로우 설명, 사용 예시, FAQ, CTA 영역
- Must-have flows: 플러그인 이해, 사용 흐름 확인, 설치 또는 문의 유도
- Nice-to-have flows: 데모 시나리오 탐색, 기존 방식과 비교 확인
- Out of scope: 실제 계정 가입 플로우, 결제 플로우, 복잡한 대시보드 구현

## 6. Conversion or Success Event

- Primary CTA or success action: 플러그인 설치 요청
- Secondary CTA: 데모 요청 또는 파일럿 논의 시작
- Expected user journey after click or action: 기능 이해 후 설치 검토 또는 팀 내 공유와 도입 상담으로 이어짐

## 7. Brand Direction

- Tone: 실무 중심, 빠른 실행, 과장 없는 설명
- Visual constraints: 화려한 마케팅 사이트보다 운영 도구처럼 신뢰감 있는 구조
- Reference sites or products: Linear, Vercel, Stripe docs 계열의 명확한 정보 구조 참고

## 8. Content and Data Inputs

- Existing copy available: NOTE.md, README.md, docs/marketplace-positioning.md
- Existing assets available: 없음
- Existing API, CMS, or data source: 없음
- Missing assets or data: 로고, 실제 사용 사례, 지표, 시각 가이드

## 9. Technical Constraints

- Preferred stack: Next.js
- Existing codebase: 현재 없음
- Authentication needs: 없음
- Analytics tools: 미정
- Deployment target: Vercel 우선 검토

## 10. Experiment Scope

- Hypothesis 1: 사용 흐름을 단계별로 시각화하면 설치 요청 전환율이 올라간다.
- Hypothesis 2: 홈페이지 전용 툴이 아니라 범용 웹 제작 플러그인이라는 점을 강조하면 관심 범위가 넓어진다.
- Segments to test: 그로스팀, 제품팀, 내부 플랫폼 운영팀

## 11. Delivery Constraints

- Deadline: 미정
- Review owner: 미정
- Approval process: 초안 검토 후 마켓플레이스 등록 문안 확정
- Non-negotiables: 범용성, 실무성, 재사용 가능성

## 12. Assumptions

- 현재 단계에서는 실제 배포 가능한 앱보다 설득력 있는 구조와 실행 예시가 더 중요하다.
- 코드베이스가 없으므로 첫 단계 산출물은 문서와 구현 계획 중심이다.
